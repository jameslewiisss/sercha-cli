# Governance

This document describes the governance model for Sercha CLI.

## Roles

### Maintainers

Maintainers have write access to the repository and are responsible for:

- Reviewing and merging pull requests
- Triaging issues
- Ensuring code quality and consistency
- Participating in project decisions
- Helping contributors

### Release Manager

The release manager (currently the project owner) is responsible for:

- Approving VERSION changes
- Coordinating releases
- Monitoring release pipelines
- Communicating release status

### Contributors

Contributors are community members who contribute to the project through:

- Code contributions
- Documentation improvements
- Bug reports
- Feature requests
- Helping other users

## Decision Making

### Routine Decisions

Day-to-day decisions (bug fixes, minor improvements) are made by individual maintainers through the PR review process.

### Significant Decisions

Significant changes (new features, breaking changes, architectural decisions) require:

1. Discussion in a GitHub issue
2. Consensus among maintainers
3. Documentation of the decision

### Disputes

If maintainers cannot reach consensus:

1. Discuss in a dedicated issue
2. Allow time for all maintainers to weigh in
3. Project owner makes final decision if needed

## Becoming a Maintainer

Contributors may be invited to become maintainers based on:

- Consistent, high-quality contributions
- Understanding of project goals and conventions
- Positive interactions with the community
- Demonstrated reliability

### Process

1. Existing maintainer nominates contributor
2. Discussion among maintainers
3. Consensus decision
4. Invitation extended

## Code of Conduct

All participants must follow the [Code of Conduct](CODE_OF_CONDUCT.md).

Violations should be reported to security@custodialabs.com.

## Changes to Governance

Changes to this governance document require:

1. PR with proposed changes
2. Discussion period (minimum 1 week)
3. Approval from project owner
